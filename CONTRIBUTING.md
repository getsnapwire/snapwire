# Contributing to Snapwire

Snapwire is the deterministic circuit breaker for AI agents. We welcome contributions that make autonomous AI safer for everyone.

## Contributor's First Day (15-Minute Onboarding)

1. **Fork & clone** the repository
2. **Install locally**:
   ```bash
   pip install -e .
   ```
3. **Run the test suite**:
   ```bash
   python -m pytest tests/ -v
   ```
4. **Start the app**:
   ```bash
   python main.py
   ```
5. **Start the Sentinel proxy** (separate terminal):
   ```bash
   python -m sentinel
   ```
6. **Run the example agent** through the proxy:
   ```bash
   OPENAI_BASE_URL=http://localhost:8080/v1 python examples/basic_agent.py
   ```
7. **Check the dashboard** at `http://localhost:5000` — you should see the intercepted tool call.

You're now running the full governance pipeline locally. Pick an issue and start contributing!

## Contributing a Sentinel Protocol Detector (3 Steps)

The highest-impact contribution is adding a new protocol detector to the Sentinel proxy. This is a 3-step process:

### Step 1: Create Your Detector Function

Add a new function in `sentinel/detector.py`:

```python
@register_protocol
def detect_my_protocol(body: dict) -> list[DetectedToolCall]:
    results = []
    # Parse the request body for your protocol's tool-call pattern
    if body.get("my_protocol_marker"):
        name = body.get("tool_name", "unknown")
        params = body.get("params", {})
        results.append(DetectedToolCall(name, params, "my_protocol", 0.9))
    return results
```

### Step 2: Register It

The `@register_protocol` decorator handles registration automatically. Your detector will be called for every request that passes through the proxy — return an empty list if the request doesn't match your protocol.

### Step 3: Add a Test

Create a test in `tests/test_sentinel.py`:

```python
def test_detect_my_protocol():
    from sentinel.detector import detect_tool_calls
    body = {"my_protocol_marker": True, "tool_name": "example", "params": {"key": "value"}}
    results = detect_tool_calls(body)
    assert len(results) == 1
    assert results[0].protocol == "my_protocol"
```

Run `python -m pytest tests/test_sentinel.py -v` to verify.

## What We're Looking For

### Sentinel Protocol Detectors

New protocol parsers for the Sentinel proxy — each one extends Snapwire's coverage to a new AI vendor or standard. See the 3-step guide above.

### Deterministic Safeguards

Hard mathematical limits that don't rely on AI judgment:

- **Loop detection rules**: New patterns for catching agent misbehavior (`src/loop_detector.py`)
- **Schema validation rules**: Per-tool JSON schemas that enforce safe parameter boundaries (`src/schema_guard.py`)
- **Blast radius rules**: New limit types for constraining agent behavior (`src/blast_radius.py`)
- **OpenClaw safeguards**: Attack detectors for real-world CVEs (`src/safeguard_openclaw.py`)

### Bug Fixes and Improvements

- Security hardening
- Performance improvements for high-frequency agent tool calls
- Dashboard UI improvements
- Documentation and examples

## AI-Generated PRs Policy

We welcome AI-assisted contributions. If your PR was partially or fully generated by an AI coding assistant:

1. **Disclose it** in the PR description (e.g., "Co-authored with GPT-4" or "Generated by Copilot")
2. **You are still responsible** for the code — review it, test it, understand it
3. **All tests must pass** — AI-generated code gets the same CI bar as human-written code
4. **Security-sensitive changes** (rules, safeguards, crypto) require manual review regardless of authorship

## Development Setup

### Prerequisites

- Python 3.11+
- PostgreSQL (or use the Docker setup)

### Local Setup

```bash
pip install -e .
export DATABASE_URL="postgresql://user:pass@localhost:5432/snapwire"
export SESSION_SECRET="dev-secret"
python main.py
```

### Docker Setup

```bash
docker-compose up
```

The app will be available at `http://localhost:5000` and the Sentinel proxy at `http://localhost:8080`.

### Running Tests

```bash
python -m pytest tests/ -v
```

## Pull Request Guidelines

1. **One feature per PR** — keep changes focused and reviewable
2. **Include tests** — especially for new deterministic rules and protocol detectors
3. **Don't break existing rules** — the 3x/30s fuse and existing schema guards must continue to pass
4. **Update documentation** — if you add a new rule type or protocol detector, document it
5. **Follow existing code style** — match the patterns in neighboring files
